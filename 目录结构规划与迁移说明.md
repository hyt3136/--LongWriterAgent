# wen_ben 项目目录结构规划与迁移说明

## 1. 目标

在不打断现有开发节奏的前提下，建立清晰、可扩展、可运维的目录结构。

约束：
1. 保持现有课程文档和代码可用。
2. 逐步迁移，避免一次性大改。
3. 适配单模型串行编排场景。

---

## 2. 推荐目录（目标态）

```text
wen_ben/
  code/                         # 可运行代码与测试
    src/wenben_engine/
    tests/
    pyproject.toml
    README.md
    .gitignore

  md/                           # 课程与章节文档（教学向）
    00_课程导航/
    01_第一章_脚手架与分层/
    ...
    12_第十二章_发布灰度与SOP/
    99_附录_模板/
    章节完成记录.md

  docs/                         # 项目文档（工程向）
    architecture/               # 架构图、设计决策、状态模型
    operations/                 # 运维SOP、发布回滚、值班手册

  artifacts/                    # 运行产物（不进源码）
    runs/                       # 执行输出、trace、临时结果

  scripts/                      # 自动化脚本（构建、校验、发布）

  LANGGRAPH_工业级从零设计蓝图.md
  课程章节规划与教课重点.md
  目录结构规划与迁移说明.md
```

---

## 3. 分层规则（必须遵守）

1. code 只放“可执行对象”：源码、测试、运行配置。
2. md 只放“课程内容”：讲解、知识点、实操与记录。
3. docs 只放“工程文档”：架构、运维、发布。
4. artifacts 只放运行产物，不放源代码。
5. scripts 只放自动化脚本，不放业务逻辑。

---

## 4. 当前已落地的优化

1. 已创建工程文档目录：docs/architecture、docs/operations。
2. 已创建运行产物目录：artifacts/runs。
3. 已创建脚本目录：scripts。
4. 已补充代码忽略规则：code/.gitignore。

---

## 5. 建议迁移步骤（低风险）

第1步（立即可做）：
1. 后续所有运行输出统一写入 artifacts/runs。
2. 新增工程说明统一写入 docs/。

第2步（后续逐步）：
1. 把与教学无关的总览文档逐步从 md 移入 docs。
2. 在 docs/architecture 中补图：主流程图、状态图、调用链图。

第3步（仓库化前）：
1. 清理 code 与 tests 下 __pycache__ 产物。
2. 保证仓库只提交源码与文档，不提交运行垃圾文件。

---

## 6. 命名规范

1. 目录用小写英文或“数字_中文章节名”二选一，不混用空格。
2. 代码文件使用 snake_case。
3. 文档文件统一前缀：
   - 01_代码解析.md
   - 02_知识点.md

---

## 7. 验收标准

满足以下即视为目录结构合格：
1. 新成员5分钟内能定位代码、课程文档、工程文档、运行产物。
2. 运行后不会在源码目录产生无关垃圾文件。
3. 章节开发与工程维护互不干扰。
