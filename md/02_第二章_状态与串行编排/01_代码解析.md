# 第二章代码解析：状态版本控制与串行编排

本章目标：在单大模型机器上，建立严格串行的编排基础，避免任何并行执行。

## 1. 新增模块一览

1. 状态Schema：code/src/wenben_engine/graph/state_schema.py
2. 状态仓库：code/src/wenben_engine/state/store.py
3. 串行运行器：code/src/wenben_engine/graph/sequential_runner.py
4. 测试：
   - code/tests/test_state_store.py
   - code/tests/test_sequential_runner.py

## 2. state_schema.py 在做什么

核心职责：定义统一 GraphState，并提供初始状态构造函数。

关键点：
1. meta 中维护 state_version。
2. control 中固定 sequential_mode=True。
3. 初始化时就把 stop_signal、retry_count等控制位放进去，避免后续节点到处判断空值。

## 3. store.py 在做什么

核心职责：提供带版本号的状态读写接口。

关键接口：
1. init_run(run_id, initial_state)
2. read_state(run_id) -> StateRecord(state, version)
3. apply_patch(run_id, expected_version, patch)
4. append_event(run_id, event)

关键机制：
1. CAS（Compare-And-Set）
   - expected_version 必须等于 current_version 才允许写。
2. 版本自动递增
   - 每次 apply_patch 后 version + 1。
3. 事件留痕
   - 每次 patch 都记录 patch_keys 和版本变化。

这为后续并发冲突处理打下基础。

## 4. sequential_runner.py 在做什么

核心职责：按顺序执行节点，且一次只执行一个节点。

执行流程：
1. 读取当前状态。
2. 调用节点 handler(state) 得到 patch。
3. 用 CAS 提交 patch。
4. 写 node_executed 事件。
5. 检查 stop_signal，若为 True 则提前停止。

这段逻辑明确保证了：
- 串行执行
- 可追踪
- 可提前终止

## 5. 测试如何验证设计

test_state_store.py 验证：
1. 正常版本写入成功。
2. 错误版本写入触发 StateConflictError。

test_sequential_runner.py 验证：
1. 节点按顺序运行。
2. stop_signal 生效后，后续节点不会运行。

## 6. 为什么本章对你很关键

你机器只有一个大模型，最怕的是流程复杂后引入伪并行逻辑导致不稳定。
本章把“只能串行”从口头约束变成了代码约束。
后续所有章节都会基于这个运行器扩展，不会走偏。
